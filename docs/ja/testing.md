# テスト戦略

## テストフレームワーク
- `pytest` を使用。
- スキーマ、diff、planner の決定的なユニットテストを優先。

## テストレイヤー
1. スキーマ解析と検証。
2. スナップショット正規化。
3. 差分計算。
4. 適用計画生成（順序と依存解決）。
5. Discord Interaction をモックしたコマンドフロー。
6. 権限ガードと確認ガード。

## 必須シナリオ
1. IDありで名称変更された対象を update/rename として検出。
2. name のみで一意一致する場合は更新。
3. name のみで複数一致する場合は検証エラー。
4. 削除差分はプレビューに表示され、確認前はハード削除されない。
5. 確認後のチャンネル削除対象は `GSM-Dustbox` へ移動される。
6. 確認後のカテゴリ削除対象は子チャンネルを `GSM-Dustbox` へ移動し、カテゴリ本体はアーカイブされる。
7. `GSM-Dustbox` がない場合は管理者のみ閲覧可能な権限で作成される。
8. ロール削除は Bot がハード削除せず、手動削除対象として報告される。
9. 全 `/schema` コマンドで非管理者実行を拒否。
10. 実行者以外の確認操作を拒否。
11. 適用前バックアップを毎回生成。
12. カテゴリ/チャンネル overwrite の追加・更新・削除差分を検出。
13. 親カテゴリ移動と並び替え差分を検出。
14. 未対応チャンネル種別で検証エラー。
15. Discord API 部分失敗を `failed[]` に分離し、成功分は `applied[]` に残す。
16. Bot 再起動で保留確認プランが失効。
17. `guild.id` 不一致の続行承認時は roles/categories/channels の同一判定を name 優先へ切り替える。
18. `/schema diff` と `/schema apply` では name 一致が曖昧でも比較を継続し、差異として報告する。
19. `bot_managed=true` のロール変更は diff/preview に表示され、apply では `skipped[]` に理由 `bot_managed_role` で記録される。
20. ロール階層制約は、Create の位置クランプ（`bot top role position - 1`）、Update/Reorder の `role_hierarchy_restriction` スキップ、diff/apply の `適用時スキップ理由` 列表示一致で検証する。

## モック戦略
- discord.py の HTTP/guild 境界をモック。
- 現在状態/目標状態は fixture スナップショット化。
- ユニットテストでは実 Discord API 呼び出しを行わない。

## CI 想定テストセット
高速デフォルト:
- 検証テスト
- 差分エンジンテスト
- プラン順序テスト
- コマンドガードテスト

拡張セット（任意）:
- コンテナ smoke test
- 一時テストギルドへの統合テスト（毎PR必須ではない）

## 受け入れ基準
- 必須シナリオをすべてカバー。
- 確認なしで apply が動かないことをテストで保証。
- 権限チェックと実行者チェックを明示的にアサート。
- 管理者限定のコマンド実行制御を明示的にアサート。
