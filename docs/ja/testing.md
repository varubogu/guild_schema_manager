# テスト戦略

## テストフレームワーク
- `pytest` を使用。
- スキーマ、diff、planner の決定的なユニットテストを優先。

## テストレイヤー
1. スキーマ解析と検証。
2. スナップショット正規化。
3. 差分計算。
4. 適用計画生成（順序と依存解決）。
5. Discord Interaction をモックしたコマンドフロー。
6. 権限ガードと確認ガード。

## 必須シナリオ
1. IDありで名称変更された対象を update/rename として検出。
2. name のみで一意一致する場合は更新。
3. name のみで複数一致する場合は検証エラー。
4. role 削除差分はプレビューに表示され、確認前は未実行。
5. 全 `/schema` コマンドで非管理者実行を拒否。
6. 実行者以外の確認操作を拒否。
7. 適用前バックアップを毎回生成。
8. カテゴリ/チャンネル overwrite の追加・更新・削除差分を検出。
9. 親カテゴリ移動と並び替え差分を検出。
10. 未対応チャンネル種別で検証エラー。
11. Discord API 部分失敗を `failed[]` に分離し、成功分は `applied[]` に残す。
12. Bot 再起動で保留確認プランが失効。

## モック戦略
- discord.py の HTTP/guild 境界をモック。
- 現在状態/目標状態は fixture スナップショット化。
- ユニットテストでは実 Discord API 呼び出しを行わない。

## CI 想定テストセット
高速デフォルト:
- 検証テスト
- 差分エンジンテスト
- プラン順序テスト
- コマンドガードテスト

拡張セット（任意）:
- コンテナ smoke test
- 一時テストギルドへの統合テスト（毎PR必須ではない）

## 受け入れ基準
- 必須シナリオをすべてカバー。
- 確認なしで apply が動かないことをテストで保証。
- 権限チェックと実行者チェックを明示的にアサート。
- 管理者限定のコマンド実行制御を明示的にアサート。
