# 差分と適用

## 差分カテゴリ
検出された変更は次のいずれかに分類:
- `Create`
- `Update`
- `Delete`
- `Move`
- `Reorder`

## 差分出力形式
主出力は人間可読の Markdown:
- 変更件数サマリ。
- リスク情報付き詳細テーブル。

詳細テーブル列:
- `action`
- `target_type`
- `target_id`
- `before_name`
- `after_name`
- `before`
- `after`
- `risk`

## 適用フロー
`/schema apply` は1コマンドで確認付き実行:
1. アップロード内 `guild.id` が現在ギルドIDと不一致か確認。
2. `guild.id` が存在し不一致なら、現在ギルドIDへ上書きするか確認。
3. アップロードスキーマを検証。
4. 差分計算しプレビュー表示。
5. 実行者向け確認ボタンを提示。
6. 承認後、バックアップ作成 -> 適用実行。

## 部分アップロードルール
- `file_trust_mode=false`（既定）では `/schema diff` と `/schema apply` のアップロードは部分指定に対応。
- パッチモードでは未指定セクション/未指定リソース/未指定フィールドは現状維持として扱う。
- パッチモードでは未記載による削除差分は発生しない。
- パッチモードではアップロードで未定義のリソースを `変更なし（ファイル未定義）` として表示する。
- guild とアップロードの両方で定義され完全一致するリソースを `変更なし（完全一致）` として表示する。
- `file_trust_mode=true` ではアップロードを完全スキーマとして扱う。
- 信頼モードでは未記載リソースに対して削除差分を生成する。
- `/schema diff` では name 一致が曖昧でも即時エラーにせず、差異として比較を継続する。
- channels の name ベース一致では、親カテゴリスコープとチャンネル種別を先に照合する。
- `bot_managed: true` のロールは diff/preview には通常表示するが、apply 実行では該当ロール操作を除外する。

## 結果返却
- diff 結果は常にダウンロード可能な Markdown ファイルとして添付する。
- apply のプレビューと実行結果レポートも Markdown ファイルとして添付する。

## 確認ルール
- アップロード内 `guild.id` が存在し現在ギルドIDと異なる場合、diff/apply 本処理の前に明示確認を要求する。
- この確認がキャンセルまたはタイムアウトした場合、コマンド実行を中止する。
- この不一致確認を承認した場合、そのリクエストの roles/categories/channels 同一判定は name 優先で処理する。
- この name 優先モードでは、roles/categories/channels の同一判定で ID へのフォールバックは行わない。
- 確認ボタン有効時間は10分。
- 実行者本人のみ確認可能。
- 実行者以外の操作は拒否。
- 有効期限切れ時は再実行が必要。

## バックアップ方針
- すべての適用前に現行構成を必ずエクスポート。
- バックアップは最終応答で添付返却。
- レスポンス処理後に永続保存しない。

## 削除方針
- 削除差分も通常の差分としてプレビュー表示。
- 明示確認後にのみ削除実行。
- 確認済みのチャンネル削除は `GSM-Dustbox` への移動として実行する。
- カテゴリ削除は子チャンネルを `GSM-Dustbox` へ移動し、カテゴリ本体は手動削除前提でアーカイブする。
- `GSM-Dustbox` がない場合は管理者のみ閲覧可能な権限で自動作成する。
- ロール削除は Bot がハード削除せず、手動クリーンアップ対象として報告する。

## 原子性と失敗時挙動
- 実行モデルはグローバルトランザクションではなくベストエフォート。
- 安全に継続可能な独立操作は続行。
- 最終レポートを以下に分離:
  - `applied[]`
  - `failed[]`
  - `skipped[]`
- 除外された bot 管理ロール操作は `skipped[]` に理由 `bot_managed_role` で記録する。

## 再起動とセッション失効
- 保留プランはメモリ内のみ。
- Bot 再起動で未承認プランは失効。
